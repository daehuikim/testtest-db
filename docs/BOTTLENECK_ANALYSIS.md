# 데이터 수집 스크립트 병목 분석 및 해결 방안

## 1. 요약

| 스크립트 | 예상 요청 수 (2024~2025) | 주요 병목 | 예상 소요 시간 (기존) |
|----------|--------------------------|-----------|------------------------|
| **auction** | 21,900회 (730일 × 30시장) | 순차 처리 + 1~2초 sleep | **5~10시간** |
| **somae** | 730회 | 순차 처리 | ~15~30분 |
| **domae** | 730회 | 순차 처리 | ~15~30분 |
| **weather** | 1~731회 | 날짜 범위 지원 시 1회 | ~1분 |

## 2. 상세 병목 분석

### 2.1 auction.py (가장 심각)

**현재 구조:**
```
for each day (730일):
    for each market (30개):
        requests.get() → 1~2초 sleep
```

**병목 원인:**
1. **이중 순차 루프**: 날짜 × 시장 = 21,900회 순차 API 호출
2. **과도한 sleep**: `random.uniform(1, 2)` → 요청당 1~2초 대기
3. **I/O 바운드**: 네트워크 대기 시간이 CPU보다 큼 → 병렬화 효과 극대화

**해결 방안:**
- **날짜별 시장 병렬화**: 같은 날짜의 30개 시장을 `ThreadPoolExecutor` 또는 `asyncio`로 동시 요청
- **sleep 최소화**: 0.2~0.5초로 축소 (API rate limit 확인 필요)
- **배치 단위 병렬**: 날짜도 5~10일씩 묶어서 병렬 처리

**예상 개선:** 21,900초 → ~1,500초 (약 **15배 단축**)

---

### 2.2 somae.py / domae.py

**현재 구조:**
```
for each day (730일):
    requests.get()  # 순차
```

**병목 원인:**
- 730회 순차 API 호출
- sleep은 없으나 순차 처리로 인한 지연

**해결 방안:**
- **날짜 배치 병렬**: 10~20개 날짜씩 `ThreadPoolExecutor`로 동시 요청
- **semaphore로 rate limit**: 동시 10~20개로 제한하여 API 과부하 방지

**예상 개선:** 730초 → ~100초 (약 **7배 단축**)

---

### 2.3 weather.py

**현재 구조:**
- 단일 API 호출로 `tm1`~`tm2` 날짜 범위 + 여러 `stn` 지원
- 2024-01-01 ~ 2025-12-31 전체를 한 번에 요청 가능할 수 있음

**병목 원인:**
- 현재 하드코딩된 짧은 기간 (2일)만 사용
- 날짜 범위가 길면 API 제한이 있을 수 있음

**해결 방안:**
- **월별 또는 분기별 배치**: API 제한 시 30~90일 단위로 분할 요청
- **병렬 배치**: 월별 요청을 병렬로 처리

## 3. 통합 아키텍처

```
run_collect_all.py
├── auction (병렬: 날짜별 시장 동시 요청)
├── somae (병렬: 날짜 배치)
├── domae (병렬: 날짜 배치)
└── weather (병렬: 월별 배치)

→ combine_raw_data.py: 날짜 기준 시계열 통합
```

## 4. 데이터 결합 구조

| 소스 | 날짜 컬럼 | 형식 |
|------|-----------|------|
| auction | 경락일시 | YYYYMMDD |
| somae | EXAMIN_DE | YYYYMMDD |
| domae | EXAMIN_DE | YYYYMMDD |
| weather | TM | YYYYMMDD |

**통합 방식:** 날짜(date)를 기준으로 모든 raw 데이터를 하나의 디렉터리/파일에 정리.  
세부 구조는 분석 목적에 따라 wide(집계) 또는 long(원본 유지) 선택 가능.
